# template/projects/scan-and-trigger/

> Canonical scaffold for the defensive automation pattern: **periodic security scan → triage → ticket → patch → re-scan → notify.**
>
> Generated by `bequite init <project> --template=scan-and-trigger [--with-wazuh]`.
> Article IX (Constitution v1.2.0) binding. `vibe-defense` Doctrine auto-loaded when `audience: vibe-handoff`.

## Default scanner stack (lite)

The lite stack runs in CI on every PR (harden-on-deploy gate) + nightly cron (drift detection). Configured in `bequite.config.toml`:

```toml
[security]
scanners = ["trivy", "semgrep", "osv", "secret-scan"]
fail_on_severity = "HIGH"
adr_override_expiry_days = 90      # override an HIGH finding via ADR; ADR auto-expires in 90d

cve_watcher_enabled = true
cve_watcher_cron = "0 9 * * *"

disclosure_timer_enabled = true     # tracks coordinated-disclosure SLAs

scan_and_trigger_target = "jira"    # jira | linear | slack | webhook
```

## Full opt-in stack (with `--with-wazuh`)

When user opts in:

```toml
[security]
scanners = ["trivy", "semgrep", "osv", "secret-scan", "nuclei", "zap"]
siem = "wazuh"                       # spins up Wazuh + Indexer + Dashboard via compose
runtime_security = "falco"           # syscall + K8s audit log monitoring
```

Heavier (Wazuh idle ~6GB RAM); appropriate for `Mode = enterprise` or projects with security teams.

## scan-and-trigger flow (the canonical operational artifact)

```
scheduler (cron / Trigger.dev / GitHub Actions / Wazuh rule)
        ↓
scanner orchestrator
  ├─ Trivy:    containers + IaC + SBOM
  ├─ Semgrep:  SAST per language
  ├─ OSV:      dep-tree CVE
  └─ secret-scan: existing hook
        ↓
triage
  ├─ deduplicate by finding-fingerprint
  ├─ severity classification
  ├─ false-positive filter (via Doctrine + project's FP allowlist)
  └─ ownership routing (CODEOWNERS)
        ↓
trigger
  ├─ Jira/Linear ticket (with CVE link + suggested patch + severity + SLA timer)
  ├─ Slack alert (#security or per-team channel)
  ├─ PagerDuty page (CRITICAL only)
  └─ auto-PR (Dependabot-style for dep bumps with known fix)
        ↓
verify
  └─ re-scan after PR/deploy; close ticket only when scanner agrees
```

## harden-on-deploy CI gate (`infra/ci/harden-on-deploy.yml`)

```yaml
name: Harden on deploy
on:
  pull_request:
    branches: [main]
jobs:
  harden:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Trivy scan
        uses: aquasecurity/trivy-action@latest
        with:
          scan-type: fs
          severity: HIGH,CRITICAL
          exit-code: 1                # block on HIGH+

      - name: Semgrep SAST
        uses: returntocorp/semgrep-action@latest
        with:
          config: p/owasp-top-ten
          severity: ERROR,WARNING
          # block on ERROR (HIGH+); WARNING is informational

      - name: OSV-Scanner
        uses: google/osv-scanner-action@latest
        with:
          scan-args: --recursive --skip-git ./

      - name: secret-scan
        run: bash skill/hooks/pretooluse-secret-scan.sh < <(git diff origin/main..HEAD)

      - name: bequite audit
        run: python -m cli.bequite.audit --ci

      - name: bequite freshness (quarterly)
        if: github.event_name == 'schedule'
        run: python -m cli.bequite.freshness --all

      - name: a11y (axe-core; frontend Doctrines)
        if: contains(github.event.pull_request.labels.*.name, 'frontend')
        run: pnpm exec @axe-core/cli https://staging.example.com
```

## Pattern 2 — harden-on-deploy (preflight gate)

PR merge → deploy pipeline runs the gate. Each finding either:
- **PASS** → deploy.
- **FAIL with active ADR override** → deploy with warning logged; ADR auto-expires in 90d; on expiry, finding re-blocks unless re-justified.
- **FAIL no override** → block deploy.

## Pattern 3 — incident-response runbook

When a Wazuh / Falco / Sigma rule fires (or a customer reports an incident):

1. **Auto-enrich**: httpx fingerprint of source IP, AbuseIPDB lookup, ASN/geo, recent CVEs for the affected service.
2. **Score**: severity × confidence × asset criticality.
3. **Contain**: firewall block / disable user / rotate creds (per `runbooks/INCIDENT_RESPONSE.md`).
4. **Investigate**: timeline, IoC search, forensic snapshot.
5. **Notify**: PagerDuty/Slack with full context bundle.
6. **Post-mortem**: template auto-generated at `.bequite/memory/incidents/<id>.md` per the configured template.

## Verification (Phase 6 gates added by Article IX)

`bequite verify` adds these gates:

1. **Lite scanner stack green** in last 24h (`evidence/<phase>/security-scan-<date>/`).
2. **No HIGH SAST findings** without active 90d-expiring-ADR-override.
3. **`bequite cve-check`** runs daily; no unhandled CRITICAL/HIGH from `cve-watcher`.
4. **Threat model exists** for every feature touching auth / regulated data / external integration.
5. **CODEOWNERS up-to-date** for security-sensitive paths (auth/, infra/, .github/, migrations/).
6. **Compliance pre-flight** complete for every active regulated Doctrine.
7. **Disclosure-timer status** clean (no red-status disclosures past SLA).
8. **(Enterprise) tabletop exercise** within last 12 months.

## Forking guidance

Common forks:

- **`scan-and-trigger-sca-only`** — lighter; only OSV + secret-scan; for libraries / packages where CI is the only deploy.
- **`scan-and-trigger-with-wazuh`** — heavier; full SIEM; for `enterprise` Mode.
- **`scan-and-trigger-air-gapped`** — for environments with no internet egress; uses local Nuclei templates + offline OSV DB snapshot.
