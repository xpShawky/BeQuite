# Multi-model planning

> Single-model planning fails predictably. The model commits to a stack early and never contradicts itself. Multi-model planning fixes that — by having two models plan independently, then comparing.

## Why two heads are better than one

Each model has signature blind spots:

- **Claude** tends to **over-engineer edge cases** (good defensiveness; bad scope creep)
- **GPT-5** tends to **under-engineer security** (good UX; bad threat-model coverage)
- **Gemini** tends to **over-trust user intent** (good speed; bad assumption-questioning)

Two models reading each other's work catches a meaningful share of these blind spots before code is written.

Veracode 2025 reports ~45% of AI-generated code samples hit OWASP Top-10 issues. Many of those are *plan-stage* failures (e.g. "store passwords in DB"). A second model reviewing the first's plan catches them.

## How it works (manual-paste mode)

This is the **default** for v0.10.5 onward. It works with your existing Claude Pro + ChatGPT Plus subscriptions. It is **ToS-clean** because you're a human pasting — no automation against subscription endpoints.

### Step 1 — Scaffold

```bash
cd ~/my-project
bequite plan --multi-model "Add SMS reminders to bookings"
```

Output:

```
docs/planning_runs/RUN-2026-05-10T15-30/
├── input_brief.md            ← what you described
├── prompts/
│   ├── plan_claude-opus-4-7.md      ← brief for Claude
│   └── plan_gpt-5.md                 ← brief for ChatGPT
├── state.json
└── receipts/                   ← will fill as you work
```

### Step 2 — Independent drafting

Open Claude (https://claude.ai or Claude Code), open `prompts/plan_claude-opus-4-7.md`, copy it into Claude. Wait for the response. Save Claude's response to `claude-opus-4-7_plan.md` in the run directory.

Now do the same with ChatGPT (https://chat.openai.com) and `plan_gpt-5.md` → `gpt-5_plan.md`.

> **Independence is the point.** Don't show one model the other's response yet. That's a different mode (Debate; see below).

### Step 3 — Compare

```bash
bequite models compare
```

BeQuite reads both plans. It parses each into structured sections (Vision / Scope / Stack / Phases / Risks / etc.). For each section, it builds a comparison row:

| Topic | Claude says | ChatGPT says | Agreement | Risk | Better option | Reason |
|---|---|---|---|---|---|---|
| Auth provider | Better-Auth | Clerk | partial | low | Better-Auth (ownership) | Aligned with Doctrine Rule 9 + ownership preference |
| State manager | Zustand | Redux Toolkit | conflict | low | Zustand | Smaller; matches Doctrine recommendation |
| Notification queue | Inngest | DIY synchronous | conflict | **HIGH** | Inngest | Article V scale-honesty: synchronous in-process emails at 5k MAU is a violation |
| ... | ... | ... | ... | ... | ... | ... |

Decisions are auto-resolved when:
- An option violates an Iron Law (Articles I-X) — auto-rejected, regardless of who proposed it.
- An option violates a loaded Doctrine rule — auto-rejected.
- An option contradicts `bequite freshness` evidence — auto-rejected.
- Both models agree — auto-accepted.

Genuine tradeoffs (where both are valid) are marked **"requires user decision"**.

### Step 4 — Merge

#### With a judge (recommended for big plans)

```bash
bequite models merge --judge claude
```

BeQuite scaffolds `prompts/merge_judge.md`. Paste it into Claude. Save the response to `merge_report.md`. Claude (acting as Judge) writes:

- Decisions accepted (with reason citing the Doctrine rule / Article / freshness fact)
- Decisions rejected (with reason citing the violated Iron Law / Doctrine / freshness)
- Decisions deferred to user (genuine preference tradeoffs the user must pick)
- Mandatory self-attestation block ("I read all peer plans... I addressed every contested topic with reasoning... I marked X topics as user-decision...")

#### Without a judge (faster — for small plans)

```bash
# Edit user_decisions.md to mark your final picks for any "requires user decision" rows
$EDITOR user_decisions.md

# Then commit
bequite models merge --confirm
```

Final plan written to `specs/<feature>/plan.md` with full provenance.

## Five collaboration modes

`--mode <mode>` switches the workflow.

### Mode 1 — Parallel (default)

Both models receive the same brief. Both draft independently. BeQuite compares + merges. **Best for early project design.**

### Mode 2 — Specialist Split

Each model owns a domain.

```bash
bequite plan --multi-model \
  --models "claude-opus-4-7=lead-architect,gpt-5=product-strategist,claude-opus-4-7=security-reviewer" \
  --mode specialist \
  "<your brief>"
```

Claude writes the architecture plan. GPT writes the product strategy plan. Claude (with security-reviewer hat) writes the security review. BeQuite stitches them.

**Best for:** larger projects where domain depth matters more than cross-domain consistency.

### Mode 3 — Debate and Merge

Round 1: parallel (each model drafts). Round 2: each model receives **the other model's plan** and is asked to identify agreement, disagreement, missing points, risky assumptions, better alternatives. BeQuite produces final based on debate outputs.

```bash
bequite plan --multi-model --mode debate "<your brief>"
```

**Best for:** contested / high-stakes decisions where independent stress-test is worth the extra round (and the extra paste-back).

### Mode 4 — Judge

Models produce parallel plans. One model is designated **Final Judge** — reviews all plans, selects best ideas, rejects weak assumptions, writes final plan. Same as `--judge X` after parallel.

### Mode 5 — Red Team

After the plan exists (any of Modes 1-4), a red-team model attacks it. Looks for:

- Security gaps (OWASP Top 10 + LLM Top 10)
- Architecture gaps (single-points-of-failure, scaling bottlenecks)
- Testing gaps (happy-path-only, missing concurrency)
- Deployment gaps (no rollback, no monitoring)
- UX gaps (missing empty/loading/error states)
- Token waste (frontier-model overspend on cheap tasks)
- Hidden assumptions ("the user has reliable internet", "the user is technical")

```bash
bequite plan --multi-model --mode red-team "<existing plan path>"
```

Output: `red_team_review.md` with severity-tagged findings (block / warn / nit).

**Best for:** vibe-defense and gov-fedramp Doctrines where adversarial review is mandatory before implementation.

## Cost expectations

- **Manual-paste mode = $0** (your subscriptions cover it).
- **Direct-API mode** (when you want automation, post-v0.11.x): $0.05-$0.30 per planning run for 2-model parallel (Claude Opus 4.7 + GPT-5). Mode 3 (Debate) costs ~1.5-2× that. Mode 4 (Judge) adds 1 judge call. Mode 5 (Red-Team) adds 1 red-team call.

Cost ceiling default: $5 per planning run (configurable via `bequite.config.toml::multi_model.max_run_cost_usd`). Hit → pause for user approval.

## What gets saved

Every multi-model run produces (in `docs/planning_runs/RUN-<datetime>/`):

```
input_brief.md             # what you described
prompts/                   # per-model prompts
  plan_claude-opus-4-7.md
  plan_gpt-5.md
  merge_judge.md           # if you ran --judge
  red_team.md              # if you ran --mode red-team
claude-opus-4-7_plan.md    # Claude's response
gpt-5_plan.md              # GPT's response
comparison.md              # decision table
merge_report.md            # judge's report
final_plan.md              # the merged plan
user_decisions.md          # your human-in-the-loop picks
red_team_review.md         # red-team findings (if applicable)
state.json                 # run state
receipts/                  # per-model receipts (signed)
```

This is your audit trail. A future engineer (or you, 6 months from now) can reconstruct exactly what was decided + why.

## Anti-patterns (what NOT to do)

- ❌ **Skipping comparison.** "I trust Claude more, just use Claude's plan" defeats the entire point of multi-model planning. The comparison surfaces what *both* models miss together.
- ❌ **Asking Claude.ai or ChatGPT.com to read each other's plans by pasting one into the other directly.** That's Mode 3 (Debate); use it intentionally, not by accident.
- ❌ **Ignoring Iron Law / Doctrine auto-rejections.** They exist because the proposal violates a binding rule. If you really want to override, file an ADR explaining why the rule shouldn't apply (rare).
- ❌ **Using consumer-subscription session cookies to drive the API directly.** Anthropic + OpenAI ToS prohibit this. Manual-paste is the ToS-clean path.
