# Starting from scratch

> You have an idea. You want to build it. BeQuite turns "an idea" into "a working, verified, deployable thing" without the half-built mess that AI coding usually produces.

## The mental model

Every project goes through 7 phases. None are skippable.

| Phase | What you produce | Time |
|---|---|---|
| **P0 — Discovery** | A short doc describing the product + audience + competitors | 5-15 min |
| **P1 — Stack** | An ADR (Architectural Decision Record) listing what tech you're using and why | 10-20 min |
| **P2 — Plan** | A spec.md + plan.md describing exactly what gets built | 15-30 min |
| **P3 — Phases** | Phase-by-phase breakdown (e.g. P5-phase-1 = foundation, P5-phase-2 = customer flow, etc.) | 5-10 min |
| **P4 — Tasks** | Atomic ≤5-min tasks per phase | 5-10 min |
| **P5 — Implement** | The actual code. One task at a time. Per-task commit + receipt. | hours |
| **P6 — Verify** | Tests pass. Playwright walks pass. axe pass. Smoke pass. | 5-15 min |
| **P7 — Handoff** | A HANDOFF.md a second engineer can run from cold | 10-15 min |

**Why this works:** every phase has an artifact, a Skeptic kill-shot question, a verification gate, and a signed receipt. Skipping a phase = breaking discipline. The agent won't let you skip.

## A worked example

Let's build something concrete: **a bookings SaaS for a hair salon**. Customers book; the salon admin manages availability.

### Phase 0 — Discovery (5-15 min)

```bash
cd ~ && bequite init salon-bookings \
  --doctrine default-web-saas \
  --mode safe \
  --scale small_saas

cd salon-bookings
bequite discover
```

The agent interviews you. Expect questions like:

- **Who's the user?** "Walk-in customers booking online instead of calling."
- **Who's the admin?** "The salon owner; one or two stylists."
- **What's the primary action a customer takes in 30 seconds?** "Pick a stylist + slot, enter name + email, confirm."
- **What's the primary admin action in 30 seconds?** "View today's bookings, mark completed, see no-shows."
- **What's NOT in scope (to keep this small)?** "No payment processing v1 (cash at salon). No SMS reminders v1. No multi-location."

Output: `docs/PRODUCT_REQUIREMENTS.md`. **Read it.** Confirm it matches your intent. Edit if needed (the file is the source of truth — if it says "no SMS" and you actually want SMS, edit it before P1).

### Phase 1 — Stack (10-20 min)

```bash
bequite decide-stack
```

The agent walks you through:

```bash
# automatically run by the agent
bequite freshness                 # checks every candidate package + price + license
                                  # (no rotted dependencies; no PhantomRaven-style hallucinations)
```

For default-web-saas, the agent will likely propose:
- **Frontend:** Next.js (App Router) + shadcn/ui + tweakcn-derived tokens.css + Tailwind v4
- **Backend:** Hono on Bun
- **Database:** Supabase (Postgres + Auth + Storage)
- **Auth:** Better-Auth (or Clerk if you prefer speed > ownership)
- **Hosting:** Vercel
- **Validation:** Zod everywhere
- **Tests:** Playwright + Vitest

It writes `.bequite/memory/decisions/ADR-001-stack.md` with the full reasoning + each Doctrine rule the choice satisfies. **Read it.** This is the only stack decision you make for this project; everything downstream assumes it.

### Phase 2 — Plan + Contracts (15-30 min)

```bash
bequite plan
```

OR (better — uses two models for cross-checked planning):

```bash
bequite plan --multi-model "Customer booking flow + admin dashboard"
# (then paste prompts into Claude + ChatGPT, save responses, run:)
bequite models compare
bequite models merge --judge claude
bequite models merge --confirm
```

Output: `specs/<feature>/spec.md` + `plan.md` describing:
- 4-6 user flows (sign-up, book, manage profile, sign-out, etc.)
- API contracts (Zod schemas shared between Next.js + Hono)
- Data model (User, Tenant, Stylist, Slot, Booking)
- RLS policies (deny-by-default per Doctrine Rule 10)
- Test strategy

### Phase 3 — Phases (5-10 min)

```bash
bequite phases
```

Decomposes the feature into committable phases:

- phase-1: Foundation (auth + RLS + tokens.css)
- phase-2: Customer flow (browse → book)
- phase-3: Admin flow (manage availability + dashboard)
- phase-4: Notifications (confirmation emails)
- phase-5: Polish + Verify

Each phase has an exit gate (specific tests that must pass) + a Skeptic kill-shot question.

### Phase 4 — Tasks (5-10 min)

```bash
bequite tasks
```

Atomic ≤5-min tasks per phase. Example tasks for phase-1:

- T-1.1: scaffold apps/web (Next.js)
- T-1.2: scaffold apps/api (Hono on Bun)
- T-1.3: drop in tokens.css with recorded font choice
- T-1.4: wire Better-Auth (email + magic link)
- T-1.5: scaffold Supabase migrations (deny-by-default RLS)
- T-1.6: hello-world `/` page
- T-1.7: CI green on hello-world

### Phase 5 — Implement (hours)

For each task:

```bash
bequite implement T-1.1
```

The agent:
1. Reads the task spec.
2. Implements the code.
3. Runs format + typecheck + tests.
4. **Verifies operational completeness** (Iron Law X: build refresh + service restart + wire-up confirmed + dependencies installed).
5. Commits with a Conventional Commits message.
6. Emits a signed receipt.
7. Marks the task done in `tasks.md`.

If anything fails, BeQuite says exactly what failed (no weasel words, no "should work"). You fix or ask the agent to retry.

### Phase 6 — Verify (5-15 min)

```bash
bequite verify
```

Runs the entire validation mesh:
- Lint + typecheck (eslint + tsc)
- Unit + component (Vitest)
- Playwright admin-walk + user-walk at viewport 360 + 1440
- axe-core accessibility gate (zero WCAG AA violations)
- Smoke (curl every public endpoint, expect status per spec)
- Secret scan + audit + freshness

All green = ready for Phase 7. Anything red = the evidence file tells you which gate failed and how to fix.

### Phase 7 — Handoff (10-15 min)

```bash
bequite handoff
```

Generates `HANDOFF.md` with:

- TL;DR for engineers (clone + boot + deploy in 4 commands)
- TL;DR for non-engineers (the vibe-handoff section — what the app does + how to operate)
- Local dev setup
- Deploy procedure
- Risk register
- Operating cost estimate ($45-90/mo at 5k MAU for a default-web-saas)
- Skeptic kill-shot ("can a different engineer ship a feature in their first week?")

## What you avoided

By going through this discipline, you skipped:

- ❌ Spending 4 hours debugging because the agent quietly said "use SQLite for write-heavy multi-tenant" (Article V scale-honesty violation — Doctrine catches it)
- ❌ A merged PR that ships an SQL injection because nobody asked the Skeptic about input validation (Doctrine Rule 11 + Skeptic gate)
- ❌ A "tests pass" report that's actually false because the dev server was never restarted (Iron Law X — operational completeness)
- ❌ A handoff doc that says "use the API" without specifying which env vars (the HANDOFF template demands them)

## Cost expectations

A small SaaS like the salon-bookings example:

- **Building it (your time):** 8-16 hours of focused work over 1-2 days.
- **AI cost (model calls):** $5-$20 in Anthropic API or $0 if you use manual-paste mode with Claude Pro + ChatGPT Plus.
- **Hosting (after launch):** $25/mo Supabase Pro + $20/mo Vercel Pro = ~$45-90/mo at 5k MAU.

**That's it.** No hidden gotchas. No "oh wait, you also need X."

## When you're stuck

```bash
bequite recover                # generates a paste-able recovery prompt
bequite audit --repo .         # constitution + doctrine drift detector
bequite freshness              # are my dependencies still fresh?
bequite verify                 # what gate is failing?
```
