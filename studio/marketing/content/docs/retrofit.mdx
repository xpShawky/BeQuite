# Retrofit BeQuite into an existing project

> Your project exists. It works (mostly). You've heard about BeQuite and want the discipline without the rewrite. **Good news: BeQuite is designed to retrofit safely.** Every step is reversible until you say otherwise.

## What "retrofitting" means

You install BeQuite's harness **alongside** your existing code. It adds:

- A `.bequite/` folder for memory + decisions + receipts.
- A `state/` folder for operational state.
- An `AGENTS.md` at the repo root.
- A `.gitignore` extension (private keys never committed).

It does **not** touch your existing source code. It does not change your build. It does not require you to migrate.

## When NOT to retrofit

Skip BeQuite if:

- The project is < 50 LOC and you'll throw it away tomorrow.
- The project is a one-off script with no expected lifetime.
- You're prototyping a 10-minute spike.

For everything else (real apps with users, business tools, anything you'd cry about losing), retrofit is worth it.

## Step 1 — Survey the existing project

Before you install BeQuite, run this checklist mentally:

- [ ] What's the **stack**? (Next.js? Rails? Django? Vue?)
- [ ] What's the **scale tier**? (solo / 1-5k / 50k / 1M+ users?)
- [ ] What **doctrine** matches? `default-web-saas` for most web apps. `cli-tool` for CLIs. `desktop-tauri` for desktop. `library-package` for libraries. (See [Doctrine authoring](/docs/doctrine-authoring) for the full list.)
- [ ] What's the **mode**? `fast` for tools/demos. `safe` for real apps (default). `enterprise` for regulated/high-stakes.
- [ ] Are there **existing docs**? (README, CONTRIBUTING, ADRs?) These become Memory Bank seeds.

## Step 2 — Install BeQuite at the repo root

```bash
cd ~/my-existing-project
bequite init . \
  --doctrine default-web-saas \
  --mode safe \
  --scale small_saas \
  --retrofit
```

The `--retrofit` flag tells BeQuite this is an existing repo. It:

1. Creates `.bequite/memory/` with the 6 Memory Bank files **pre-populated from your README + CONTRIBUTING + package.json + existing ADRs** (if present).
2. Creates `.bequite/keys/` + `.bequite/.keys/` keypair.
3. Creates `state/recovery.md` with a "Migrating from non-BeQuite project" status.
4. Creates `.bequite/memory/decisions/ADR-RETRO-001-baseline.md` capturing the as-is state.
5. **Does not modify your `src/`, `app/`, `pages/`, etc.** Your code is untouched.

After running, `git status` should show the new `.bequite/`, `state/`, `AGENTS.md`, and a `.gitignore` extension. **Review and commit.** This is your baseline.

## Step 3 — Capture the as-is in the Memory Bank

The Memory Bank's six files describe what the project *is right now*. BeQuite pre-populates them, but they need a human pass to be honest:

```bash
# read each in your editor:
.bequite/memory/projectbrief.md       # the elevator pitch
.bequite/memory/productContext.md     # who it's for + why it exists
.bequite/memory/systemPatterns.md     # how it's built (architecture)
.bequite/memory/techContext.md        # the actual stack + versions
.bequite/memory/activeContext.md      # what's in flight RIGHT NOW
.bequite/memory/progress.md           # short evolution log
```

Edit each. Be honest:

- Is the architecture you wish for actually the architecture you have? **Document the as-is, not the wish.**
- Are there known-tech-debt patches that don't follow Doctrine rules? **List them in systemPatterns.md as "known violations" with a target ADR.**

> **Article VI honest reporting** applies. If `techContext.md` says "we use TypeScript everywhere" but 30% of `src/` is JavaScript, write that down. The agent reads this on every session and assumes it's true.

## Step 4 — Run the audit (find your skeletons)

```bash
bequite audit --repo .
```

This runs the audit rule packs against your existing code:

- Article IV violations (secrets in source? `.env*` reads in code?)
- Doctrine violations (hardcoded `bg-purple-500`? Card nesting? gray-on-color text?)
- Article V scale-honesty violations (synchronous in-process at >50k tier?)
- Article VII hallucination defense (imports not in lockfile?)

Expect findings. **That's the point.** The audit is your retrofit's todo list.

For each finding:

- **block** severity → file an issue + an ADR explaining the plan to fix.
- **warn** severity → track in `docs/risks.md`.
- **nit** severity → either fix in a polish pass or accept as design.

## Step 5 — Run freshness probe (find your rotted deps)

```bash
bequite freshness
```

Checks every package in your lockfiles:

- Is the version still maintained? (last-commit < 6 months)
- Are there unfixed criticals on OSV?
- Is the vendor pricing still what you assumed?
- Are any packages in the PhantomRaven supply-chain incident list?

Findings here are usually *more urgent* than audit findings — security vulnerabilities and hostile takeovers happen in dependencies first.

## Step 6 — Add a feature the BeQuite way (validates the retrofit)

Now that BeQuite is installed and your as-is is documented, exercise the workflow on a small feature.

```bash
bequite discover
# (interview about the feature — small one to start)

bequite plan
# (or: bequite plan --multi-model for cross-checked planning)

bequite tasks
bequite implement <task-id>
bequite verify
```

The agent will:

- Read your existing Memory Bank (so it understands the as-is).
- Plan the feature *against your existing architecture*, not from scratch.
- Implement in your existing src/ tree using your existing conventions.
- Verify with whatever test runner you already use (BeQuite detects pytest, vitest, jest, etc.).

If the implementation passes, your retrofit succeeded. You now have:
- A receipt for the change.
- A spec/plan in `specs/`.
- An updated `progress.md`.
- Operational completeness verified.

## Step 7 — Optional: gradually adopt Doctrines

You don't have to fix every audit finding immediately. Pick a Doctrine rule per week:

- Week 1: Adopt Doctrine Rule 1 (tokens-only design) — refactor hardcoded values into `tokens.css`.
- Week 2: Adopt Doctrine Rule 11 (Zod validation everywhere) — wrap one API endpoint per day.
- Week 3: Adopt Doctrine Rule 12 (Playwright walks) — write the admin walk first, user walk second.
- Etc.

Each adoption = one ADR + one set of code changes + one passing audit run.

## When retrofit goes wrong

If `bequite init . --retrofit` fails or produces something off:

1. **It can't break your existing code.** It only adds new folders.
2. **You can roll back:** `git clean -fd .bequite state` + `rm AGENTS.md` + revert `.gitignore` changes.
3. **Tell me what happened:** the error message + the project's stack + your shell. I (or the BeQuite community) can fix the retrofit detection logic.

## Common retrofit pain points (real examples)

| Symptom | Cause | Fix |
|---|---|---|
| `bequite audit` finds 200+ warn-severity findings | Long-running project with accumulated tech debt | Don't try to fix all at once; pick the top 3 block-severity items + ADR them; track the rest in risks.md |
| Stack doesn't match a built-in Doctrine | Custom stack (e.g. PHP + jQuery + MariaDB) | Fork the closest Doctrine: `cp skill/doctrines/default-web-saas.md .bequite/doctrines/my-stack.md` + edit |
| `bequite verify` fails because there's no Playwright | You don't have e2e tests yet | Add `tests/walkthroughs/admin-walk.md` + `user-walk.md` per Doctrine Rule 12; install Playwright; rerun |
| Freshness probe flags 12 deps as stale | Project was last updated 2 years ago | Don't bulk-update; pin one dep at a time + run tests; ADR each major-version bump |
